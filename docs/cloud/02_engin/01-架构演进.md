# 架构演进 
## 第一次演进 拆分部署
初始架构：单机架构
初始访问链：浏览器 -> DNS服务器 -> web应用 -> 数据库
服务部署：web应用 + 数据库部署在同一台服务器中
场景：随着用户的访问量增加，Tomcat和数据库之间的资源竞争，单机的性能不足

解决方案：
- 拆分部署：将tomcat和数据库分开部署。

 
目标：解决web应用和数据库之间的资源竞争问题，集群化数据库进一步增强了数据库的抗压能力。

## 第二次演进 引入缓存
初始架构：web应用和数据库分离
初始访问链：浏览器 -> DNS服务器 -> web应用服务器 -> 数据库服务器
服务部署：web应用单机部署，数据库单机部署 
场景：用户量增加，并发增高，反馈网页响应速度太慢。数据库读压力太大。

解决方案：
- 引入多级缓存：web层引入应用缓存，数据层引入redis（集群，哨兵，主从）
- 数据层优化点：redis做集群化，刷盘策略，一致性问题，存储持久化，重写策略
- 关注新的问题：击穿，雪崩，穿透。双写一致性。

目标：引入缓存层，给数据库分摊并发读压力。进一步增强了数据层的抗压能力。

## 第三次演进 引入负载均衡
初始架构：web应用单机，数据库单机，引入redis抗并发读
初始访问链：浏览器 -> DNS服务器 -> web应用服务器 -> redis集群 —> 数据库服务器
服务部署：web应用单机部署，缓存层集群化部署，数据库单机部署
场景：用户量持续增加，网站在做活动时，时长有上万并发，web应用经常处理不过来请求，tcp连接池溢出。安全隐患，web服务器经常被黑客攻击。

解决方案：
- 引入负载均衡层，做反向代理，将请求均匀分发到web集群中
- 将web服务器集群化，部署到多台服务器上。
- 用户session共享问题，上传下载问题

目标：引入负载均层，并将web服务器集群化，将流量分摊到多台服务器上

## 第四次演进 读写分离
初始架构：web应用集群，数据库单机，引入redis抗并发读
初始访问链：浏览器 -> DNS服务器 -> web应用服集群 -> redis集群 —> 数据库服务器
服务部署：web应用集权化部署，缓存层群化部署，数据库单机部署
场景：用户量增大，读并发持续的增高，数据库单机成为瓶颈
解决方案：
- 将数据库做成集群化，主从架构，读写分离，从库负责读


## 第五次演进 分库分表
分库：将一份完整的数据，分散到不同的库里，不同的库部署在不同的机器，主要是分摊单库压力 
分表：把一份完整的数据分散到不同的表中，主要为了解决单表数据量过大的读写压力
